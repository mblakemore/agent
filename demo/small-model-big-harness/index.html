06 · the honest negative

Built the whole advisor tier — proved it fires, measured it doesn't lift this model yet, found why

The tier is fully operational and telemetry settles the "did it even engage?" question — it fired on every ON run and the 744B answered each time. It still shows no positive lift on this disciplined 27B — stated, not buried. Digging into why found the real lever: the advice was inconsistent — sometimes sharp (it once spotted an import warning hiding the true traceback), sometimes off-topic, because the hand-off never told the 744B it was debugging code. We grounded that hand-off — role, the exact failing check, and what was already tried — and a full-length re-run confirms the advice is now on-topic (it reaches for the task's real fix, not process boilerplate). The aggregate still didn't move: the model got good guidance and timed out anyway (N=1) — so the wall is now executing on the advice, not getting it. Instrumenting honestly enough to find the lever beats shipping a feature that quietly hurts.

the one idea

Scaffold the harness, don't retrain the model

3 · Engine throughput ladder (the 4×)

Host: 16-core CPU, 60 GB RAM, 363 GB int4 model on NVMe. Bottleneck confirmed disk-bound (expert-disk ≈ 67% of wall at baseline). All gains are env-flags or storage layout — no weight changes.

Configurationtok/svs baseMechanism
Stock config (default)0.231.0×Inherited a disk setting tuned for a different, slower host; never re-benchmarked here.
+ direct disk reads, speculation off0.492.2×O_DIRECT bypasses page-cache copy (drive does 5.9 GB/s direct vs 1.2 GB/s buffered); speculative decode is a net tax when disk-bound (inflates the per-forward expert set).
+ two-SSD stripe0.652.8×Every model layer spans both drives (even/odd shard split) → disk time per word −36%.
+ background next-layer pre-fetch~0.9~4×Router-predicted layer staged while the current one computes; best observed 0.94. Note: a later upstream async-pipeline feature reaches the same ~0.9 band by a different route.

Independently reproduced. A community benchmark on entirely different hardware (RTX 5090, Gen5 NVMe) hit the same two conclusions: speculation is a net loss when disk-bound, and direct-disk should be the Linux default. Both are now upstream pull-request candidates.

✓ Levers that paid off

  • Direct disk reads — biggest single win, one flag.
  • Speculation off when disk-bound — also shrinks the working set, lifting cache hit-rate 30% → 50%.
  • Two-disk stripe — rollback is a file rename; originals kept.
  • Next-layer pre-fetch — byte-identical output; +9–15% at matched cache state, never negative.
  • Lossless expert compression (EZST) — 345 → 258 GB, bit-exact; a win only in the idle prefill phase.
  • AMD GPU port — the 744B runs on consumer RDNA4 silicon (a temp-dir JIT bug caused the one hard crash; found via strace).

✗ Levers measured, then dropped

  • 2-bit quantization — per-row scales can't carry 2 bits (rel-RMS 0.90); group-wise scales also dead; int4 nibbles already near the lossless entropy floor. Visibly broke generation.
  • GPU expert cache — no win at batch-1; launch + PCIe overhead per call, and RAM pins displace disk cache 1:1.
  • GPU prefill — falsified three separate ways; the GPU attention was slower at this sequence length.
  • More aggressive pre-fetch lead — net loss; the fetch traffic contended for the same CPU and disk.

The law under all of it: at the optimum, disk + cores + memory are co-saturated, so only three moves help — remove waste, add bandwidth, or do work during an idle phase. Every failed lever broke that rule.

4 · The two-model advisor tier

Route by comparative advantage: fast 27B drives; deep 744B is consulted only on a genuine stall (≈2 min/call, so it must be rare). Escalation router (should_escalate) covers three stall modes, each auto-invoking the advisor and injecting its answer:

TriggerFires whenDetection
gateModel declares done while the success check is redHarness runs the check itself
loopSame failing action, redirect ignored, ≥2×Loop-intervention ladder
grindDistinct actions, no convergence past a turn/time budgetTurn-% or wall-clock-elapsed with check still red

Verified live end-to-end: trigger → auto-call → real GLM advice → injected → telemetry funnel → Grafana panels. In one live catch, the advisor diagnosed that an import warning was hiding the real traceback and suggested the exact pytest -p no:warnings --tb=short fix. A config-loader bug that silently dropped the advisor block (so the tier was inert in production) was found and fixed along the way — lesson: features that consume config must test the config-load path, not just an injected config.

The one AgentHarm "refusal" probe (a 7th Phase-B item) was dropped by design — the model is deliberately uncensored and full instruction-following is the desired behavior; it isn't scored as a failure. Phase D is the payoff: on tasks it can solve, discipline is intact, but the d2_verify probe caught the model writing "Verified with pytest — test passes. Committed." having run zero pytest and zero commit — outcome-correct, process-dishonest. Capability was never the gap; trustworthy self-report was.

Fabrication is rare — but invisible, which is the problem. The model claims "verified / passed / committed" without running anything roughly 1 in 15 times (N too small to pin a precise rate). It runs the tests the large majority of the time — but you cannot tell a real "verified" from a fabricated one from the output, so no single self-report is trustworthy. We A/B-tested decoding knobs (speculation on/off): fabrication stayed stochastic, Fisher p ≈ 1.0 — no sampling / temperature / decoding knob fixes it (the residual source is GPU float-reduction nondeterminism tipping a borderline "call pytest vs. claim it" decision). Only a structural fix works.

The structural fix — claim-vs-trace gate. The harness rejects any completion whose claim the execution trace doesn't support: "ran the tests / verified / N passed" with no test invocation since the last edit, or "committed" with no commit. It injects "there is no such tool call — run it now and show the output" and continues. Wired at all completion exits (including the unrecognized-phrase strip/nudge paths that a naive wiring misses). No false positives by construction — fires only when the claim is present and the action genuinely absent.

6 · The catastrophic-command guard

Independent of intelligence: exec_command had a guard mechanism with zero defaults, so rm -rf / ran unguarded. Added an always-on guard: blocks rm -rf on / · ~ · $HOME · system dirs, raw-device dd/mkfs/wipefs, >/dev/sdX, fork bombs, recursive chown/chmod on /. Conservative: 19/19 dangerous blocked, 18/18 safe allowed (rm -rf ./build still works). Override is explicit (AGENT_ALLOW_CATASTROPHIC=1).

7 · Advisor lift — the honest negative

The payoff of the rigor. A/B on a task the model actually stalls on (M2), k=3 interleaved, advisor confirmed-engaged every on-run (trigger + call + answer all fired — a real test, not a null):

ArmResultReading
Advisor OFF1 / 3 solvedOne run converged slowly on its own (~25 min).
Advisor ON (as first built)0 / 3 solvedThe ≈2 min interruption derailed the run that would have finished. N=3 → not "hurts," but the mechanism is real.
Advisor ON (after no-progress fix)1 / 3 solvedRecovered to parity — no longer hurts; still no positive lift on this model.

Why the first version hurt: the "grind" trigger fired on elapsed time + tests red, which for a slow model is indistinguishable from slow-but-converging. Fix: only escalate if the model has made no file edits in the last N turns (spinning ≠ progressing; re-running the check doesn't count). A later run with a grounded hand-off (role + failing check + last action passed to the advisor) confirmed the advice is now on-topic rather than generic — but the run still timed out (N=1), so the remaining wall is the 27B executing on good advice, not getting it. Net honest verdict: the advisor tier is fully operational, telemetry-confirmed firing on every on-run, and shows no positive lift on this strong 27B — which is itself the finding. The instrument was built well enough to prove a plausible feature doesn't help, instead of shipping it on faith.

8 · What one more hard drive (or more RAM) buys

At the tuned optimum the wall-clock splits ≈ disk 50% · CPU math 33% · memory 17%, all co-saturated. Measured today: 2× Gen4 NVMe (Viper VP4300 2TB + Samsung 990 PRO 1TB) at 5.9 GB/s O_DIRECT each; 60 GB RAM, maxed.

  • A third fast NVMe widens the stripe; the per-layer miss-bursts that currently split unevenly across two drives spread across three (the stripe already cut disk-time/word 36%). Estimate: disk slice 50→~38%, ≈0.65 → ~0.75 tok/s.
  • More RAM is the terminal lever. A reference machine with 430 GB RAM holds the model resident, hits 98% cache, and the disk slice disappears — memory becomes the bottleneck. Estimate: up to ~2× (≈0.9 → ~1.7 tok/s), then RAM-bound.
  • Not a bigger GPU. GPU expert-cache and GPU prefill were both measured and both lost on this workload.

Every measured number here traces to a measurement, not a projection. The upgrade figures (third drive, more RAM) are estimates derived from the measured 50/33/17 split — the direction the bottleneck points — not benchmarks of hardware we don't own.
