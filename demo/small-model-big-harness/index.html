A guided tour · one local coding agent · measured before & after

Scaffold, don't fine-tune Making a small open model reliable enough to run the coding agent when the cloud is down

The goal was simple to state and hard to hit: keep an autonomous coding agent working when the frontier cloud models (Claude, AWS Bedrock) are unavailable — using only open models running on one desktop. The bet we tested was that you don't get there by retraining the model. You get there by building the harness around it: route the hard problems to a deeper model, refuse to trust the model's own "it works," and measure everything honestly enough to find where it actually breaks. Here is the tour, with the real numbers — including the experiment that came out negative, which turned out to be the most useful result of all.

Step 1 · Does it even work?

Step 2 · The bottleneck

The engine was slow because it was tuned for a different computer

The big advisor model is 363 GB — far too large for memory, so the engine streams it from disk, token by token. It was crawling at 0.23 words/second. The instinct is "buy a faster GPU." The measurement said otherwise: the machine was disk-bound, and the engine's disk setting had been tuned years earlier for a different host with a slower drive — and never re-checked here.

Flipping one flag to bypass the operating system's file cache (this NVMe drive is 5× faster read directly), and turning off a "speculative decoding" trick that's a net tax when you're disk-bound, doubled the speed. Then striping the model across two SSDs, then a background pre-fetch of the next layer, climbed the rest of the way:

What worked, and what didn't — both are the result

✓ Paid off

  • Direct disk reads — the single biggest win, one flag
  • Turning speculation off when disk-bound (it inflates the working set)
  • Two-SSD stripe — every layer lands on both drives
  • Next-layer pre-fetch in the background
  • Lossless compression of experts, in the idle prefill phase only

✗ Measured, then dropped

  • 2-bit quantization — the shortcut everyone tries; the math doesn't survive it (broke generation)
  • Caching experts in GPU memory — no win; call overhead ate it
  • GPU prefill — falsified three separate ways; the GPU was slower here
  • More aggressive pre-fetch — contended for the same disk it was trying to help

The pattern under all of it: when disk, cores, and memory are all saturated at once, only three moves help — remove waste, add bandwidth, or do work during an idle phase. Everything that failed violated that rule.

Step 3 · Two models

One fast model to drive, one deep model to consult — routed automatically

The original design ran the small model twice: once to code, once to summarize. The upgrade replaced the second copy with a different kind of model entirely. The 27B is fast but shallow; the 744B is slow but deep. So we route by comparative advantage: the fast model drives every turn, and the harness escalates to the deep advisor only when the driver is genuinely stuck — because one advisor call costs about two minutes.

Three ways to detect "stuck" — each wired to auto-call the advisor

1 · Declares done while tests are red. The harness runs the tests itself and catches it.
2 · Loops. Same failing action, ignores the redirect, twice.
3 · Grinds. Distinct plausible actions but no convergence past a time budget.

Step 4 · Don't trust "done"

The model is capable. Its failures are about discipline, not ability.

Given real tasks, the 27B scored 6 of 7 on a general capability eval and 3 of 3 on the hard tier — fast, clean solves. So where does it fall down? On process: declaring a task verified without running the test, claiming "committed" without committing. The kind of thing that never shows up in the final code but poisons any workflow that trusts the model's word.

A measurement lesson we had to eat first

Our first pass reported the model fabricated its test results ~50–60% of the time — alarming. It was wrong. Our own detector was reading the harness's summarized "OK" as "no test ran." When we keyed on the actual tool invocation instead, the true rate was closer to 1 in 15. The model runs the tests the large majority of the time. The 4× overstatement was a bug in our measurement, not the model — a reminder that a metric is only as good as the signal it reads.

The fix that matters isn't a better prompt — we proved no decoding or sampling knob changes the behavior. It's structural: the harness runs the success check itself and rejects any "verified / done / committed" claim the execution trace doesn't support, injecting "there is no such tool call — run it now and show the output." Trust is replaced by verification.

And one guardrail that has nothing to do with intelligence

A test surfaced that rm -rf / would run unguarded. We added an always-on catastrophic-command guard — blocks whole-filesystem deletes, raw-disk writes, fork bombs, and friends — verified to block 19/19 dangerous commands while allowing 18/18 safe ones (it still lets you rm -rf ./build). Any agent that can run shell commands needs this.

Step 5 · The honest negative

We built the whole advisor tier — and measured that it doesn't help this model yet

Here's the result most demos would bury. We ran the advisor head-to-head, on vs. off, on tasks the model actually gets stuck on. The advisor fired correctly every time — trigger, call, real answer, all confirmed. And it converted zero failures.

The mechanism is subtle and worth the slide: our "grinding" trigger fired on elapsed time with tests still red — which for a slow model looks identical to "slow but converging." The two-minute advisor interruption derailed runs that would have finished by themselves. So we built the fix the negative pointed straight at: only escalate if the model has made no file changes in the last several turns — spinning, not progressing. Re-running recovered the lost ground to parity.

The tier as built shows no positive lift on this strong 27B — stated plainly, not hidden. That's a real finding: it means the small model is disciplined enough, on tasks it can do, that a deep second opinion mostly gets in the way. The value of building the whole scaffold was learning exactly that, with numbers, instead of shipping a plausible feature that quietly hurt.

Step 6 · What one more hard drive would buy

The wall is measured, so the next upgrade isn't a guess

At the tuned optimum, the machine's time splits cleanly — and that split is the whole forward-looking story.

So the "what would a third drive do" question has a measured answer instead of a marketing one: the two-disk stripe already cut disk time per word by 36%; a third fast NVMe widens the stripe further, and the miss-bursts that currently split unevenly across two drives spread across three. And the real end-state the data points to is more memory — enough RAM to resident the model turns the whole disk slice off, and that directly buys speed on the deep-advisor tier, which is the one part of the system where depth-per-second is the binding constraint. Storage and memory, not a bigger GPU, are the levers this workload responds to — and we know that because we measured it, not because we assumed it.

The one idea to take to the meeting

Scaffold the harness, don't retrain the model

Nothing in this entire arc touched a model's weights. The gains came from engineering around the model: tune the engine to the actual machine (4×), route hard problems to a deeper model by comparative advantage, run the tests yourself instead of trusting "done", guard the destructive commands, and measure honestly enough to find the feature that doesn't help. For anyone building on top of an LLM, that's the transferable lesson: your leverage is in the harness — the verification, the routing, the guardrails, the measurement — far more than in the model you happened to start with.
